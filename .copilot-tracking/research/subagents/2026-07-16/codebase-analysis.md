### Socket.IO Events

**Client → Server:**
- `place_tile(payload: PlaceTilePayload, ack)` — ack returns `PlaceTileAck`
- `remove_tile(payload: RemoveTilePayload, ack)` — ack returns `RemoveTileAck`
- `pointer_move(payload: PointerMovePayload)` — fire and forget

**Server → Client:**
- `session_snapshot(payload: SessionSnapshotPayload)` — on connect
- `tile_placed(payload: TilePlacedPayload)` — broadcast
- `tile_removed(payload: TileRemovedPayload)` — broadcast
- `pointer_update(payload: PointerUpdatePayload)` — all except sender
- `client_joined(payload: ClientJoinedPayload)` — broadcast
- `client_left(payload: ClientLeftPayload)` — broadcast

### Key Types (contracts.ts)

## 4. Write Path: place_tile

`apps/server/src/index.ts` lines ~490–540:

```
Client → socket.emit('place_tile', payload, ack)
         ↓
index.ts: isPlaceTilePayload(payload) guard
         ↓
loadSessionRecord(sessionId)      ← reads canvases + tiles from DB
         ↓
validatePlacement(shape, transform, tiles, defaultBounds)  ← domain logic
         ↓
  [invalid] → invokeAckSafely(ack, { rejected: true, reason })
  [valid]   ↓
persistTilePlacement({ sessionId, payload, placedBy })
         ↓
  db.transaction:
    pg_advisory_xact_lock(hashtext(sessionId))   ← per-canvas serial lock
    getNextOpSeq(tx, sessionId)                  ← MAX(op_seq) + 1
    INSERT INTO tiles ...
    INSERT INTO operation_log (op_seq, op_type='tile_placed', payload={tileId,...payload})
    UPDATE canvases SET updated_at = now()
    SELECT tiles WHERE canvas_id = sessionId      ← re-fetch full state
    return { opSeq, session, ack, event }
         ↓
invokeAckSafely(ack, result.ack)                 ← ack to placing client
io.to(sessionId).emit('tile_placed', result.event) ← broadcast to room
persistSnapshotIfNeeded(sessionId, opSeq, session) ← every 25 ops
```

### Write Path: remove_tile

Similar flow, same advisory lock, `DELETE FROM tiles`, `INSERT INTO operation_log` with op_type='tile_removed'.

**If tile not found:** transaction returns `{ removed: false }` (no log entry written, opSeq consumed but not persisted).

## 5. Read Path: Session State on Connect

`apps/server/src/index.ts` → `initializeParticipantPresence` → `loadSessionReplayRecord`:

```
socket connect
  ↓
loadSessionReplayRecord(sessionId)
  ↓
  getLatestSnapshot(sessionId)    ← SELECT FROM snapshots ORDER BY op_seq DESC LIMIT 1
  ↓
  loadSessionRecord(sessionId)    ← INSERT canvases ON CONFLICT DO NOTHING
                                     SELECT canvases, tiles, participants, max(op_seq)
  ↓
  listOperationsAfter(sessionId, snapshotOpSeq)  ← ops with op_seq > snapshot.opSeq
  ↓
  replay: apply each operation to snapshot tiles
  ↓
socket.emit('session_snapshot', { session, clients, lastOpSeq })
```

**Note:** `loadSessionRecord` reads the `tiles` table directly (authoritative live state). The snapshot + replay path is used by `loadSessionReplayRecord` for the connect event. The `tiles` table is the ground truth; `operation_log` is used for replay between snapshots.

## 6. Existing Versioning / Sequence Fields

Already present:
- `operation_log.op_seq` — per-canvas integer counter, computed as `MAX(op_seq) + 1` inside advisory-locked transaction
- `UNIQUE(canvas_id, op_seq)` — duplicate sequence impossible at DB level
- `PlaceTileAck.opSeq` — returned to placing client on success
- `TilePlacedPayload.opSeq` — broadcast with each placement event
- `TileRemovedPayload.opSeq` — broadcast with each removal event
- `SessionSnapshotPayload.lastOpSeq` — sent on connect
- `snapshots.op_seq` — snapshot indexed by the op_seq at capture time

**What's missing for full idempotency:** No client-supplied operation ID (`idempotencyKey` or similar) that would allow a client to detect and skip duplicate retries. Currently if a client retries a `place_tile` after a network drop, a second tile will be created with a different UUID.

## 7. Repository Layer Key Functions

**File:** `apps/server/src/db/repository.ts`

| Function | Purpose |
|----------|---------|
| `loadSessionRecord(sessionId)` | Upsert canvas (onConflictDoNothing), fetch tiles+participants+lastOpSeq |
| `loadSessionReplayRecord(sessionId)` | Snapshot + ops replay to reconstruct state |
| `markParticipantJoined(sessionId, clientId, joinedAt)` | Upsert participant (onConflictDoUpdate) — already idempotent |
| `markParticipantLeft(sessionId, clientId, leftAt)` | UPDATE participants SET left_at |
| `listActiveParticipants(sessionId)` | SELECT WHERE left_at IS NULL |
| `persistTilePlacement({ sessionId, payload, placedBy, tileId?, createdAt? })` | TX: lock + opSeq + INSERT tile + INSERT op_log + UPDATE canvas |
| `persistTileRemoval({ sessionId, payload, removedBy })` | TX: lock + opSeq + DELETE tile + INSERT op_log + UPDATE canvas |
| `saveSnapshot(sessionId, opSeq, session)` | INSERT INTO snapshots |
| `getLatestSnapshot(sessionId)` | SELECT latest snapshot by op_seq DESC |
| `listOperationsAfter(sessionId, opSeq)` | SELECT op_log WHERE op_seq > N |
| `pruneRetention({ operationCutoffMs, snapshotCutoffMs })` | DELETE old ops + snapshots |

**Notable:** `persistTilePlacement` accepts optional `tileId` and `createdAt` params (L255) — partial idempotency scaffolding already exists. The tile `id` is client-assigned (not DB-generated), so a deterministic `tileId` would make place idempotent against the DB `UNIQUE(canvas_id, op_seq)` constraint combined with a passed `tileId`.

## 11. Domain Logic Summary

### math2d.ts (`apps/server/src/domain/math2d.ts`)
Pure 2D vector math: `vec2`, `add`, `sub`, `scale`, `dot`, `len`, `normalize`, `perp`, `rotate`, `lerp`, `clamp`, `shortestAngleDelta`, `easeOutCubic`, `hash2`.

### placementSolver.ts (`apps/server/src/domain/placementSolver.ts`)
SAT (Separating Axis Theorem) collision detection for tile polygons.
- `MAX_GROUT_GAP = 0.22` — max allowed edge gap
- Canvas bounds: `minX=-5.2, maxX=5.2, minY=-3.4, maxY=3.4` (from contracts.ts comment)
- Returns `ValidationResult { valid, correction, penetration, reason }`
- Rejection reasons: `'out-of-bounds (...)'`, `'overlap (...)'`, `'gap too large (...)'`
- These map via `toRejectReason()` to `'OUT_OF_BOUNDS'`, `'OVERLAP'`, `'GAP_TOO_LARGE'`, `'PLACEMENT_REJECTED'`

## 13. Concurrency Model

- **Per-canvas advisory lock:** `pg_advisory_xact_lock(hashtext(sessionId))` acquired inside every write transaction before computing `op_seq`. This serializes all writes to a given canvas at the Postgres level.
- **In-memory sessions map:** `sessions: Map<string, AuthoritativeSessionState>` in index.ts — ephemeral, lost on restart. On reconnect, `loadSessionReplayRecord` rebuilds from DB.
- **opSeq gap between in-memory and DB:** `applyPlaceTile`/`applyRemoveTile` (in-memory) increment `state.lastOpSeq` independently from DB writes. The DB `persistTilePlacement` recomputes from `MAX(op_seq)`. These two counters are separate — the in-memory one is for local ordering before the DB is involved.

## Key Findings for Issue #13 (Revisioning & Idempotency)

### Already in place
1. `op_seq` sequence exists with `UNIQUE(canvas_id, op_seq)` DB constraint
2. `PlaceTileAck`, `TilePlacedPayload`, `TileRemovedPayload`, `SessionSnapshotPayload` already carry `opSeq`
3. `persistTilePlacement` accepts optional `tileId` — tile UUID is client-assignable
4. Advisory locks prevent concurrent duplicate op_seq
5. `markParticipantJoined` uses `onConflictDoUpdate` (idempotent)
6. `loadSessionRecord` uses `onConflictDoNothing` for canvas (idempotent)

### Gaps / what needs adding for idempotency
1. No client-supplied idempotency key for `place_tile`. If a client retries after an ACK timeout, a second tile record is created.
2. `persistTileRemoval` doesn't carry `tileId` → can produce duplicate `op_seq` on retry (current behavior: tile not found → `{ removed: false }`, no op_log written, but opSeq counter was bumped in-memory).
3. No deduplication index on `operation_log` by client-assigned tile operation identity.
4. REST endpoints for sessions/tiles documented in contracts.ts but not implemented in index.ts — could be needed for idempotent HTTP layer.

### Clarifying questions for issue #13
1. Does idempotency mean: client can re-send the same `place_tile` with the same tile UUID and get back the same `opSeq` without a double-write? If so, the approach is: add a `tileId` field to `PlaceTilePayload` and use `INSERT INTO tiles ON CONFLICT (id) DO NOTHING`.
2. Does revisioning mean: client must provide `lastOpSeq` when sending mutations, and the server rejects out-of-order operations? Or is it just for client state reconciliation via the existing `opSeq` return values?
3. Should the REST endpoints (POST /sessions, etc.) be implemented as part of this issue, or only the Socket.IO path?
4. Should `op_seq` be exposed as a query parameter for `GET /sessions/:sessionId` to allow delta-fetching?
